## Products and Plans

### Products

A **Product** represents your API, service, or application that you want to monetize. Each product has:

- **Product Reference** (`prd_...`) - Unique identifier for your product
- **Name** - Human-readable name
- **Description** - What the product does

Products are created in the SolvaPay dashboard and represent the service you're protecting. New
products are available for SDK integration by default and are listed in the SolvaPay marketplace.

```typescript
// All endpoints for this product share the same paywall rules
const payable = solvaPay.payable({
  product: 'prd_myapi', // Your product reference
})
```

### Plans

A **Plan** defines a pricing model and access tier for a product. Plans are embedded inside products — each product has its own set of plans. Every plan includes:

- **Plan Reference** (`pln_...`) - Unique identifier
- **Type** - `recurring`, `usage-based`, `one-time`, or `hybrid`
- **Price** and **currency**
- **Features** - Provider-defined feature flags
- **Status** - `active`, `inactive`, or `archived`

```typescript
// Protect endpoints with a specific plan requirement
const payable = solvaPay.payable({
  product: 'prd_myapi',
  plan: 'pln_premium', // Users need this plan to access
})
```

### Product-Plan Relationship

- **One product can have multiple plans** - Different tiers (Free, Pro, Enterprise)
- **Plans are embedded in products** - Created and managed as part of the product
- **One default plan per product** - Used as the fallback for MCP tool access and sign-ups
- **Paywall checks** - Verify user has the required plan for the product

## Plan Types

SolvaPay supports four plan types:

| Type | Description | Best For |
|------|-------------|----------|
| **Recurring** | Fixed price billed on a cycle (monthly, yearly, etc.) | SaaS subscriptions |
| **Usage-Based** | Pay-per-use tracked by a [meter](/meters/overview) | APIs, token-based services |
| **Hybrid** | Recurring base fee + usage-based overage | Enterprise SaaS with variable usage |
| **One-Time** | Single purchase, no recurring billing | Digital goods, lifetime access |

Usage-based and hybrid plans reference a **meter** via `meterId` to track consumption and enforce limits. See [Plans documentation](/plans/overview) for full details on each type.

## Meters

**Meters** are standalone metric definitions that track usage independently of products, plans, and purchases. A meter defines *what* to measure (e.g. API requests, tokens, tool calls), and **meter events** are the raw data points recorded against those definitions.

- **Default meters** (`api_requests`, `requests`, `api_calls`, `tokens`) are auto-created for every provider
- **MCP tool meters** (`tool:{toolName}`) are auto-created when MCP servers discover tools
- **Custom meters** can be created for any business-specific metric

Plans link to meters to enable usage-based billing and limit enforcement. The meter event timeseries is the source of truth for all usage queries.

See [Meters documentation](/meters/overview) for details.

## Customer References

### What is a Customer Reference?

A **Customer Reference** (`customerRef`) is a unique identifier for a user in your system. It's used to:

- Track usage per customer
- Check purchase status
- Enforce usage limits
- Create payment intents

### How Customer References Work

1. **Customer Creation**: When a user first interacts with your protected endpoints, SolvaPay automatically creates a customer record
2. **External Reference**: You can link SolvaPay customers to your user IDs using `externalRef`
3. **Customer Lookup**: SolvaPay uses the customer reference to check purchases and track usage

```typescript
// Ensure customer exists (creates if doesn't exist)
const customerRef = await solvaPay.ensureCustomer(
  'user_123', // customerRef (your user ID)
  'user_123', // externalRef (same or different)
  {
    email: 'user@example.com',
    name: 'John Doe',
  },
)
```

### Customer Reference in Requests

For HTTP adapters, pass the customer reference via headers:

```typescript
// Express.js example
app.post(
  '/tasks',
  payable.http(async req => {
    const customerRef = req.headers['x-customer-ref']
    // Business logic here
  }),
)
```

For Next.js, the customer reference is automatically extracted from authentication:

```typescript
// Next.js automatically extracts from auth middleware
const result = await checkPurchase(request)
// result.customerRef is automatically set
```

## Paywall Protection Flow

### How Paywall Protection Works

When you wrap your business logic with `payable()`, SolvaPay automatically:

1. **Extracts Customer Reference** - From headers, auth tokens, or request context
2. **Checks Purchase Status** - Verifies if customer has required plan
3. **Checks Usage Limits** - Verifies if customer is within usage limits
4. **Executes Business Logic** - If checks pass, runs your function
5. **Tracks Usage** - Records the usage for billing/analytics
6. **Returns Paywall Error** - If checks fail, returns error with checkout URL

### Paywall Flow Diagram

```
Request → Extract Customer Ref → Check Purchase
                                    ↓
                              Has Purchase?
                                    ↓
                              Yes → Check Usage Limits
                                    ↓
                              Within Limits?
                                    ↓
                              Yes → Execute Business Logic
                                    ↓
                              Track Usage → Return Result
                                    ↓
                              No → Return Paywall Error
                                    ↓
                              (with checkout URL)
```

### Paywall Error Response

When a paywall is triggered, SolvaPay returns a structured error:

```typescript
{
  error: 'PaywallError',
  message: 'Purchase required',
  checkoutUrl: 'https://checkout.solvapay.com/...',
  // Additional metadata for custom UI
}
```

You can customize the error handling:

```typescript
try {
  const result = await handler(req)
  return result
} catch (error) {
  if (error instanceof PaywallError) {
    // Custom paywall handling
    return res.status(402).json({
      error: error.message,
      checkoutUrl: error.checkoutUrl,
    })
  }
  throw error
}
```

## Purchase Lifecycle

### Purchase States

Purchases can be in different states:

- **Active** - Purchase is active and can be used
- **Cancelled** - Purchase is cancelled but still active until end date
- **Expired** - Purchase has expired
- **Past Due** - Payment failed, purchase is past due

### Purchase Management

```typescript
// Check purchase status
const customer = await solvaPay.getCustomer({ customerRef: 'user_123' })
const purchases = customer.purchases

// Cancel renewal
await solvaPay.cancelRenewal({
  customerRef: 'user_123',
  purchaseRef: 'pur_...',
})
```

### Free Tier

All products support a **free tier** with limited usage:

- **Free Tier Limits** - Set in SolvaPay dashboard (e.g., 100 calls/month)
- **No Purchase Required** - Free tier works without payment
- **Automatic Upgrade Prompt** - When limits are exceeded, users see checkout URL

## Authentication Flow

### How Authentication Works

SolvaPay SDK integrates with your existing authentication system:

1. **Extract User ID** - From auth tokens, headers, or session
2. **Map to Customer Reference** - User ID becomes customer reference
3. **Sync Customer Data** - Email, name, etc. synced to SolvaPay
4. **Check Purchase** - Verify purchase status

### Authentication Adapters

SolvaPay provides adapters for common auth systems:

```typescript
// Supabase adapter
import { SupabaseAuthAdapter } from '@solvapay/auth/supabase'

const adapter = new SupabaseAuthAdapter({
  supabaseUrl: process.env.SUPABASE_URL,
  supabaseKey: process.env.SUPABASE_ANON_KEY,
})

// Extract user ID from Supabase JWT
const userId = adapter.getUserId(request)
```

### Next.js Authentication

Next.js helpers automatically extract user info:

```typescript
import { getAuthenticatedUser } from '@solvapay/next'

// Automatically extracts userId, email, name from Supabase JWT
const user = await getAuthenticatedUser(request)
// { userId: '...', email: '...', name: '...' }
```

## Usage Tracking

### Meter-Based Tracking

SolvaPay tracks usage through **meter events** — timestamped data points stored in a MongoDB timeseries collection. Events are recorded automatically when you use `payable()`, and can also be recorded manually.

Usage is tracked:

- **Per Meter** - Each metric has its own event stream (e.g. `api_requests`, `tool:search`)
- **Per Customer** - Usage is scoped to individual customers
- **Per Provider** - Multi-tenant isolation ensures providers only see their own data

### Automatic Tracking

When you use `payable()`, each request automatically records a meter event. MCP tool calls through the SolvaPay proxy are also tracked automatically against `tool:{toolName}` meters.

### Manual Usage Tracking

Record events directly for custom metrics:

```typescript
// Track custom usage via SDK
await solvaPay.trackUsage({
  customerRef: 'user_123',
  meterName: 'api_requests',
  value: 1,
  properties: {
    endpoint: '/api/v1/search',
  },
})
```

### Usage Limits

Usage-based and hybrid plans enforce limits through meters:

- **Hard caps** - The plan's `limit` field sets a maximum per billing period
- **Free units** - The plan's `freeUnits` defines included usage before billing starts
- **Overage** - Hybrid plans can allow overage beyond the limit at a specified rate

Limit checks query the meter event timeseries in real time — `sum(meterName, customerId, periodStart, now)` is compared against the plan's hard cap. When limits are exceeded, requests are denied with a paywall response that includes a checkout URL for upgrading.
