## Onboarding a New Customer[​](#onboarding-a-new-customer "Direct link to Onboarding a New Customer")

**You:** Create a customer for [jane@acme.com](mailto:jane@acme.com) with the name "Jane Smith"

**Assistant:** I'll create that customer for you.

*Calls `create_customer` with `email: "jane@acme.com"`, `name: "Jane Smith"`*

> The customer has been created:
>
> * **Reference:** `cus_A1B2C3D4`
> * **Email:** [jane@acme.com](mailto:jane@acme.com)
> * **Name:** Jane Smith

***

**You:** Make sure we have a customer for [bob@startup.io](mailto:bob@startup.io) — create one if they don't exist

**Assistant:** I'll use the ensure tool to check.

*Calls `ensure_customer` with `email: "bob@startup.io"`, `name: "Bob"`*

> A customer for [bob@startup.io](mailto:bob@startup.io) already exists:
>
> * **Reference:** `cus_X7Y8Z9W0`
> * **Email:** [bob@startup.io](mailto:bob@startup.io)
>
> No duplicate was created.

The `ensure_customer` tool is idempotent — it's safe to call repeatedly with the same email. It will return the existing customer if one exists, or create a new one if not.

## Setting Up an Agent with Pricing[​](#setting-up-an-agent-with-pricing "Direct link to Setting Up an Agent with Pricing")

**You:** Create an AI agent called "support-bot" for handling customer support, then set up a free starter plan and a $49/month pro plan

**Assistant:**

*Calls `create_agent` with `name: "support-bot"`, `description: "Customer support automation"`, `categories: ["support", "automation"]`*

*Calls `create_plan` with `name: "Starter"`, `type: "usage-based"`, `price: 0`, `features: ["100 free requests/month", "Community support"]`*

*Calls `create_plan` with `name: "Pro"`, `type: "recurring"`, `price: 4900`, `currency: "USD"`, `interval: "month"`, `features: ["Unlimited requests", "Priority support", "Custom integrations"]`*

> I've created the agent and both plans:
>
> **Agent:** support-bot (`agt_P9Q0R1S2`)
>
> | Plan    | Reference      | Price     | Type        |
> | ------- | -------------- | --------- | ----------- |
> | Starter | `pln_F1G2H3J4` | Free      | Usage-based |
> | Pro     | `pln_K5L6M7N8` | $49/month | Recurring   |

## Processing a Refund[​](#processing-a-refund "Direct link to Processing a Refund")

**You:** Refund $25 from transaction txn\_D7E8F9G0 — the customer was double-charged

*Calls `create_refund` with `sourceTransactionId: "txn_D7E8F9G0"`, `amount: 2500`, `reason: "Double charge"`*

> A partial refund of **$25.00** has been processed for transaction `txn_D7E8F9G0`. The refund reference is `txn_H1I2J3K4`.

For a full refund, simply omit the `amount` parameter.

## Usage-Based Billing[​](#usage-based-billing "Direct link to Usage-Based Billing")

**You:** Record that customer cus\_X7Y8Z9W0 made a successful API call to agent agt\_P9Q0R1S2

*Calls `record_usage` with `customerRef: "cus_X7Y8Z9W0"`, `agentRef: "agt_P9Q0R1S2"`, `outcome: "success"`, `action: "api_call"`*

> Usage event recorded successfully.

***

**You:** Check if that customer has hit their usage limits

*Calls `check_limits` with `customerRef: "cus_X7Y8Z9W0"`, `agentRef: "agt_P9Q0R1S2"`*

> The customer has used **87 of 100** free requests this period. They have **13 requests remaining** before they'll need to upgrade.

***

**You:** Record 5 usage events in bulk for today's batch

*Calls `record_bulk_usage` with an array of 5 events*

> 5 usage events recorded successfully.

The `record_bulk_usage` tool accepts an array of events, each with `customerRef`, `agentRef`, and `outcome`. Use this for batch processing instead of making individual calls.
