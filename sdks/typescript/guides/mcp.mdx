## Overview

This guide shows how to monetize an MCP server you host yourself using `@solvapay/server`. The SDK is framework-agnostic -- it works with `@modelcontextprotocol/sdk`, FastMCP, or any other MCP implementation.

If you want SolvaPay to handle authentication, billing, and paywall enforcement for you via a managed proxy, see [Hosted MCP Pay](/mcp-pay/overview) instead.

### What the SDK provides

| Capability | SDK Function | Description |
| --- | --- | --- |
| Paywall protection | `payable.mcp()` | Wraps tool handlers with limit checks and usage tracking |
| Self-service tools | `registerVirtualToolsMcp()` | Adds `get_user_info`, `upgrade`, and `manage_account` tools |
| OAuth bridge helpers | `createMcpOAuthBridge()` | Registers well-known endpoints and attaches `req.auth` |

## Installation

Install the SolvaPay server SDK alongside your MCP framework of choice:

```bash
npm install @solvapay/server
```

The SDK has no dependency on any particular MCP framework. Install whichever MCP library you use separately (e.g. `@modelcontextprotocol/sdk`, `fastmcp`).

## SDK Setup

### Initialize SolvaPay

Create a shared config module that the rest of your server imports:

```typescript
import { createSolvaPay, createSolvaPayClient } from '@solvapay/server'

const apiClient = createSolvaPayClient({
  apiKey: process.env.SOLVAPAY_SECRET_KEY!,
  apiBaseUrl: process.env.SOLVAPAY_API_BASE_URL,
})

export const solvaPay = createSolvaPay({ apiClient })

export const productRef = process.env.SOLVAPAY_PRODUCT_REF!

export const payable = solvaPay.payable({
  product: productRef,
})
```

### Optional: bootstrap product + plans + MCP config

If you want a one-call setup for an MCP integration, use `bootstrapMcpProduct`:

```typescript
const bootstrap = await solvaPay.bootstrapMcpProduct({
  name: 'Docs Assistant',
  originUrl: 'https://origin.example.com/mcp',
  freePlan: {
    name: 'Free',
    freeUnits: 1000,
  },
  paidPlans: [
    { key: 'pro', name: 'Pro', price: 2000, currency: 'USD', billingCycle: 'monthly' },
  ],
  tools: [
    { name: 'list_docs', planKeys: ['free'] },
    { name: 'deep_research', planKeys: ['pro'] },
    { name: 'health_check', noPlan: true },
  ],
})

console.log(bootstrap.product.reference)
console.log(bootstrap.mcpServer.mcpProxyUrl)
```

For enterprise workflows, you can still use separate endpoints for product, plans, and MCP server
management.

Field naming for bootstrap payloads:
- Use `originUrl` for the origin MCP endpoint.
- Use `freePlan` to customize the implicit free plan created by bootstrap.
- Use `paidPlans` to define paid plans (`key`, `name`, `price`, `currency`, optional billing fields).
- For tool mapping, prefer `planIds` as canonical persisted field; bootstrap also accepts
  `planRefs` and `planKeys`.

`createSolvaPay()` also accepts `apiKey` directly if you don't need a custom client:

```typescript
const solvaPay = createSolvaPay({
  apiKey: process.env.SOLVAPAY_SECRET_KEY,
})
```

### Customer identity

Every paywall check requires a **customer reference** -- a stable identifier for the user making the tool call. For MCP, the SDK can read this from `extra.authInfo`:

```typescript
const getCustomerRef = (_args: Record<string, unknown>, extra?: McpToolExtra) => {
  const customerRef = extra?.authInfo?.extra?.customer_ref
  return typeof customerRef === 'string' && customerRef.trim() ? customerRef : 'anonymous'
}
```

`payable.mcp()` defaults to this `extra.authInfo.extra.customer_ref` path automatically, so you only need a custom `getCustomerRef` when you want custom identity rules.

## Protecting Tools

### Wrap handlers with `payable.mcp()`

Write your business logic as a plain async function that receives tool arguments and returns a result object. Then wrap it with `payable.mcp()`:

```typescript
async function createTask(args: { title: string; description?: string }) {
  const task = { id: crypto.randomUUID(), ...args, createdAt: new Date().toISOString() }
  return { success: true, task }
}

const protectedCreateTask = payable.mcp(createTask, { getCustomerRef })
```

When `protectedCreateTask` is called:

1. `getCustomerRef` extracts the customer identity from the tool arguments
2. The SDK calls SolvaPay's limits API to check the customer's plan and usage
3. If within limits, your business logic runs and usage is tracked
4. If over limits, the adapter returns a paywall response with a checkout URL
5. The result is automatically wrapped in MCP `content` format (`{ content: [{ type: 'text', text: '...' }] }`)

You do **not** need to catch `PaywallError` yourself -- the MCP adapter handles it and returns a structured error response that AI clients can act on.

### Build a handler map

A common pattern is to build a map of tool names to handlers, then dispatch in your MCP framework's `tools/call` handler:

```typescript
const toolHandlers: Record<string, (args: Record<string, unknown>) => Promise<unknown>> = {
  create_task: payable.mcp(createTask),
  get_task: payable.mcp(getTask),
  list_tasks: payable.mcp(listTasks),
}
```

### Different plans per tool

Create multiple `payable` instances to assign different plans:

```typescript
const freeTier = solvaPay.payable({ product: productRef, plan: 'pln_free' })
const proTier = solvaPay.payable({ product: productRef, plan: 'pln_pro' })

const toolHandlers = {
  list_tasks: freeTier.mcp(listTasks),
  create_task: proTier.mcp(createTask),
}
```

## Virtual Tools

Virtual tools give users self-service access to their account status, upgrade options, and account management -- the same tools that Hosted MCP Pay provides automatically.

### Register virtual tools (recommended)

```typescript
await solvaPay.registerVirtualToolsMcp(server, {
  product: productRef,
})
```

This one call registers all self-service tools directly on an `McpServer`.

| Tool | Description |
| --- | --- |
| `get_user_info` | Returns current user profile and purchase status |
| `upgrade` | Returns available plans and checkout URLs |
| `manage_account` | Returns a secure link to the customer portal |

Virtual tools are **not** paywall-protected and are **not** usage-tracked.

### Advanced: manual registration

Use `getVirtualTools()` only if you need custom MCP server plumbing:

```typescript
const virtualTools = solvaPay.getVirtualTools({
  product: productRef,
  getCustomerRef,
  exclude: ['manage_account'],
})
```

## Authentication via OAuth Bridge

MCP clients (Cursor, Claude Desktop, etc.) authenticate via OAuth 2.0. Use the SDK helper to:

- serve the two well-known discovery endpoints
- decode bearer JWTs into MCP `authInfo`
- attach `req.auth` for Streamable HTTP transport
- return `401` + `WWW-Authenticate` when auth is missing/invalid

```typescript
import { createMcpOAuthBridge } from '@solvapay/server'

app.use(
  ...createMcpOAuthBridge({
    publicBaseUrl: process.env.MCP_PUBLIC_BASE_URL!,
    apiBaseUrl: process.env.SOLVAPAY_API_BASE_URL || 'https://api.solvapay.com',
    productRef: process.env.SOLVAPAY_PRODUCT_REF!,
    mcpPath: '/mcp',
    requireAuth: true,
  }),
)
```

If you need the raw metadata payloads outside middleware, use:

- `getOAuthProtectedResourceResponse(publicBaseUrl)`
- `getOAuthAuthorizationServerResponse({ apiBaseUrl, productRef })`

## Complete Example

This example uses `@modelcontextprotocol/sdk` with Express for the HTTP transport. The same SDK calls work with any MCP framework -- only the tool registration and HTTP layer differ.

```typescript
import 'dotenv/config'
import express from 'express'
import { randomUUID } from 'node:crypto'
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp'
import { StreamableHTTPServerTransport } from '@modelcontextprotocol/sdk/server/streamableHttp'
import { isInitializeRequest } from '@modelcontextprotocol/sdk/types'
import { z } from 'zod'
import { createMcpOAuthBridge, createSolvaPay, createSolvaPayClient } from '@solvapay/server'

const app = express()
app.use(express.json())

const productRef = process.env.SOLVAPAY_PRODUCT_REF!
const apiClient = createSolvaPayClient({ apiKey: process.env.SOLVAPAY_SECRET_KEY! })
const solvaPay = createSolvaPay({ apiClient })
const payable = solvaPay.payable({ product: productRef })

app.use(
  ...createMcpOAuthBridge({
    publicBaseUrl: process.env.MCP_PUBLIC_BASE_URL!,
    apiBaseUrl: process.env.SOLVAPAY_API_BASE_URL || 'https://api.solvapay.com',
    productRef,
    mcpPath: '/mcp',
    requireAuth: true,
  }),
)

function createMcpServer() {
  const server = new McpServer({ name: 'my-mcp-server', version: '1.0.0' }, { capabilities: { tools: {} } })

  void solvaPay.registerVirtualToolsMcp(server, { product: productRef })

  server.registerTool(
    'create_task',
    {
      description: 'Create a new task',
      inputSchema: { title: z.string() },
    },
    payable.mcp(async ({ title }: { title: string }) => ({ success: true, task: { id: randomUUID(), title } })),
  )

  return server
}

const sessions: Record<string, { transport: StreamableHTTPServerTransport }> = {}

app.post('/mcp', async (req, res) => {
  const sessionId = (req.headers['mcp-session-id'] as string) || ''
  let transport = sessionId ? sessions[sessionId]?.transport : null

  if (!transport && isInitializeRequest(req.body)) {
    const server = createMcpServer()
    transport = new StreamableHTTPServerTransport({
      sessionIdGenerator: () => randomUUID(),
      onsessioninitialized: sid => {
        sessions[sid] = { transport: transport! }
      },
    })
    await server.connect(transport)
  }

  if (!transport) {
    return res.status(400).json({ error: 'Missing or invalid MCP-Session-Id' })
  }

  await transport.handleRequest(req, res, req.body)
})
```

For the full working example with SSE streaming, session cleanup, and a test script, see the [`mcp-oauth-bridge` example](https://github.com/solvapay/solvapay-sdk/tree/main/examples/mcp-oauth-bridge) in the SDK repository.

## Environment Variables

| Variable | Required | Description |
| --- | --- | --- |
| `SOLVAPAY_SECRET_KEY` | Yes | Your SolvaPay API secret key (starts with `sk_`) |
| `SOLVAPAY_API_BASE_URL` | No | API base URL (defaults to `https://api.solvapay.com`) |
| `SOLVAPAY_PRODUCT_REF` | Yes | Product reference for paywall checks and OAuth DCR |
| `MCP_PUBLIC_BASE_URL` | Yes | Your server's public origin (used in well-known endpoints) |
| `MCP_PORT` | No | Port to listen on (default: `3004`) |
